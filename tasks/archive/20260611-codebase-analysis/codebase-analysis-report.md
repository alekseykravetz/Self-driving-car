## 🧱 3. Class & Module Patterns

### 3.1 Plain Classes vs Custom Elements vs Free Functions

- **Plain classes:** dominant unit (~85% of files define one global class, e.g. `Car`, `World`, `NeuralNetwork`, `Polygon`).
- **Custom elements (`extends HTMLElement`): 5 files** — `ts/store/storeManagerPanel.ts`, `ts/panels/viewControlsPanel.ts`, `ts/panels/toolbarPanel.ts`, `ts/world-editor/editors/worldEditor.ts`, `ts/ai-training/trainingManagerPanel.ts`.
- **Free-function modules:** `ts/math/utils.ts`, `ts/utils.ts`, and `ts/types.ts` (interfaces/options only).
- Files generally stick to **one top-level class per file**.

```ts
// Plain class (ts/car/car.ts)
class Car {
  x: number;
  y: number;
  speed: number;
  damaged = false;
  fitness = 0;
  // owns its own draw(ctx)
}
```

```ts
// Custom element (ts/panels/toolbarPanel.ts)
class ToolbarPanelElement extends HTMLElement {
  /* ... */
}
customElements.define('toolbar-panel', ToolbarPanelElement);
```

## 🔄 4. State & Data Flow

No Redux/Context/store framework. State flows through the global scope, the `StoreManager` singleton, `localStorage`, and instance fields.

### 4.1 In-Memory / Instance State

```ts
// ts/car/car.ts
this.speed = 0;
this.damaged = false;
this.fitness = 0;
```

### 4.2 Shared / "Global" State

- Pages share the active world/car via the **`StoreManager` singleton** (`getActiveWorld()` / `getActiveCar()`).
- The old globals `world` / `carInfo` are **gone from runtime** but still **listed in `eslint.config.mjs`'s `allowedUnusedVars`** (`"world"`, `"carInfo"`) — leftover allowance.
- Dependency injection is **mixed**: most classes receive deps via constructor, but the panels and `simulator`/`race` reach directly for `StoreManager` and `localStorage`.

### 4.3 Persistence (`localStorage`)

| Key                                        | Value              | Writer                                                                    | Reader                                         |
| ------------------------------------------ | ------------------ | ------------------------------------------------------------------------- | ---------------------------------------------- |
| `bestPool`                                 | `CarInfo[]` (JSON) | `storageManager.ts`, `simulator.ts`, `race.ts`, `trainingManagerPanel.ts` | `storageManager.ts`, `trainingManagerPanel.ts` |
| `world`                                    | World JSON string  | `world-editor/editors/worldEditor.ts`                                     | `simulator.ts`, `race.ts`, `worldEditor.ts`    |
| `bestBrain` / `bestBrains` / `bestCarInfo` | legacy brain blobs | (read/removed only)                                                       | `storageManager.ts` (migration/cleanup)        |
| `markerThreshold`                          | number string      | `car/controls/markerDetector.ts`                                          | `markerDetector.ts`                            |
| `store:activeWorld` (`ACTIVE_WORLD_KEY`)   | filename string    | `storeManager.ts`                                                         | `storeManager.ts`                              |
| `store:activeCar` (`ACTIVE_CAR_KEY`)       | filename string    | `storeManager.ts`                                                         | `storeManager.ts`                              |

**Inconsistency flagged:** the `world`/`bestPool` keys use **raw string literals** scattered across 4 files, while `store:*` keys are centralized as `StoreManager` static constants. `bestBrain*` keys exist only for legacy migration/cleanup.

### 4.4 Async / Loading Flow

- Startup: pages `await StoreManager.init()` → `fetch('/store/manifest.json')` → fetch each world/car.
- Error handling is **consistent inside `StoreManager`** (`if (!resp.ok) throw`), but the per-file `world`/`bestPool` reads use bare `localStorage.getItem(...)` + `JSON.parse` with no guard in several spots.

```ts
// ts/store/storeManager.ts
const resp = await fetch('/store/manifest.json');
if (!resp.ok) throw new Error(`manifest fetch failed: ${resp.status}`);
```

## ⚠️ 9. Anti-Patterns & Problem Areas

### 9.1 God Classes / Oversized Files (verified)

```
ts/ai-training/simulator.ts            623  (render + state + input + localStorage)
ts/ai-training/trainingManagerPanel.ts 558  (UI + persistence + pool mgmt)
ts/games/race.ts                       448  (render + localStorage)
ts/world-editor/world.ts               421
ts/world-editor/editors/worldEditor.ts 366  (also a custom element)
```

### 9.2 Duplicate Code

- JSON-vs-legacy parsing duplicated across `WorldLoader`, `CarLoader`, `StoreManager`.
- `localStorage.setItem("bestPool", JSON.stringify(...))` repeated in 4 files (`storageManager`, `simulator`, `race`, `trainingManagerPanel`).
- Projection/`draw` math repeated between `Simulator`, `Race`, `Camera`, `MiniMap` (per memory + draw count of ~28).

### 9.3 Global-Scope Coupling

- Panels and `simulator`/`race` reach directly for `StoreManager` and `localStorage` rather than receiving them.
- Hidden load-order dependencies: all classes are globals loaded via ordered `<script>` tags.
- Stale allowance for removed globals `world`/`carInfo` in `eslint.config.mjs`.

### 9.4 Inconsistent Error Handling

- Only **15 `try` blocks** total across the source. `StoreManager.init` throws on bad fetch; loaders and direct `localStorage`/`JSON.parse` reads frequently have **no error path** (silent `null` or unguarded parse).

### 9.5 Legacy / Deprecated Patterns

- Dual file formats still supported (pure JSON **and** legacy `World.load({...})` / `let carInfo = {...}`).
- `saves/` retains old wrapper-format files alongside pure-JSON `store/` assets.
- Orphaned compiled JS without TS source: `js/camera_new_ai_ver.js`, `js/shared/sharedInit.js`.
- **`any` usage is minimal — 4 occurrences in 2 files**, all defensible:
  - `ts/sound.ts` ×3: `(window as any).webkitAudioContext` (vendor-prefix fallback).
  - `ts/store/storeManager.ts` ×1: `(data as any).markings` (untyped parsed JSON).

### 9.6 Dead & Unused Code

- `js/camera_new_ai_ver.js` and `js/shared/sharedInit.js` ship without TS source; `ts/shared/` is empty — likely dead/stale output.
- `allowedUnusedVars` in ESLint config still lists removed globals.
